# Relay Workflow Companion — Design Spec

**Date:** 2026-06-18
**Status:** Draft

---

## Overview

Relay gains three new capabilities that together make it a full workflow companion for any AI agent on any team:

1. **Workflow config store** — a team's standards doc is read by an agent (LLM-backed), config fields are extracted and submitted via MCP, and stored in Relay's `globalState` after the user confirms in VS Code. Any agent on the machine reads the confirmed config via `get_workflow_config`.

2. **Skill registry** — agents submit abstract markdown skills via MCP. Relay stores them and installs agent-appropriate versions on all detected agents after user confirmation. A skill submitted by Claude Code gets installed on Cursor, Gemini, and Codex too.

3. **Generalized dev-flow skill** — a workflow skill (in the abstract skill format) covering the full development lifecycle: issue pickup, alignment, doing the work, PR flow, AI review loop, and reviewer path. Generated by an agent from a team's standards doc — not hardcoded in the extension.

**The extension never parses natural language or generates skill content.** That is left to LLM-backed agents. The extension stores, translates, distributes, and installs.

Review pages (reviewer brief, self-review notes, PR explanations) are plain `.relay` pages created via the existing `create_page` tool. No special infrastructure.

---

## Goals

- Any AI agent can submit a workflow config extracted from a standards doc via MCP
- Any AI agent can submit a workflow skill that gets installed on all other agents
- The user always reviews and confirms before anything is saved or installed
- Skills are agent-agnostic: written once in markdown, installed in each agent's native format
- The extension proposes skill installation automatically on activation, chained after MCP setup
- Review and explanation pages use the existing `create_page` tool throughout

## Architecture

Two new services alongside existing ones:

```
┌──────────────────────────────────────────────────────────────┐
│  Webview (sandboxed HTML/CSS/JS)                             │
│  src/webview/sidebar/  src/webview/page/                     │
│         ↕ postMessage                                        │
├──────────────────────────────────────────────────────────────┤
│  Extension Host (Node.js)                                    │
│  PortalViewProvider  PageViewPanel                           │
│         ↕ method calls                                       │
├──────────────────────────────────────────────────────────────┤
│  Services                                                    │
│  DataService   FaviconService   PageReader                   │
│  WorkflowConfigService          SkillRegistry                │
│         ↕                                                    │
│  VS Code globalState   workspace fs                          │
└──────────────────────────────────────────────────────────────┘
         ↑ HTTP JSON-RPC 2.0
  McpServer (127.0.0.1:3333 by default)
```

**WorkflowConfigService** owns all reads and writes to `relay.workflowConfig` and manages pending review state. No other class touches it directly.

**SkillRegistry** owns `relay.skills`, validates skill frontmatter, drives agent-format installation via `AgentAdapter`, and manages pending skill review state. No other class installs skill files directly.

## Storage

| `globalState` key | Type | Contents |
|---|---|---|
| `relay.workflowConfig` | `WorkflowConfig` | Team config submitted by agent and confirmed by user |
| `relay.skills` | `Skill[]` | Workflow skills submitted by agent and confirmed by user |
| `relay.workflowSkillDismissed` | `boolean` | Dismissed flag for the skill install activation prompt |

```typescript
interface WorkflowConfig {
  slack: {
    status: string;   // e.g. "#zdev-status"
    general: string;
    weekly: string;
    pulse: string;
    deploy: string;
  };
  language: string;   // e.g. "ro" or "en"
  githubOrg: string;
  prAccount: string;
  identity?: {        // filled by agents on first run via set_workflow_config
    githubUsername: string;
    currentRepo: string;
  };
}

interface Skill {
  name: string;         // kebab-case
  description: string;
  content: string;      // full markdown including frontmatter
  agents: string[];     // ['all'] or specific agent ids
  version: number;
  installedAt: number;  // unix ms
}
```

## New MCP Tools

Five new tools bring the total from 11 to 16.

### `submit_workflow_config`
**Args:** `config` (object, partial `WorkflowConfig`)
**Behavior:** Merges the proposed config with any existing `relay.workflowConfig`, saves to a pending state, fires a VS Code confirmation prompt (see Confirmation UX). Returns `{ status: "submitted" }` immediately — does not block. If the user confirms, the merged config is written to `relay.workflowConfig`. If dismissed, the pending state is cleared without affecting the existing config.

Agents use this both for initial setup (extracting from a standards doc) and for self-updating identity fields (`identity.githubUsername`, `identity.currentRepo`) on first run.

### `get_workflow_config`
**Args:** none
**Returns:** The current `WorkflowConfig`, or JSON-RPC error `-32603` with message `"Workflow config not configured"` if none exists.

### `add_skill`
**Args:** `name` (string), `content` (string — full markdown with frontmatter), `description` (string, optional — overrides frontmatter description if provided)
**Behavior:** Validates frontmatter (name and description required). Saves to a pending state, fires a VS Code confirmation prompt. Returns `{ status: "submitted" }` immediately. On user confirm: stored in `relay.skills` (incrementing `version` if name already exists), then installed on all configured agents whose id matches the skill's `agents` field.

### `list_skills`
**Args:** none
**Returns:** Array of `{ name, description, agents, version, installedAt }` for all stored skills.

### `remove_skill`
**Args:** `name` (string)
**Returns:** `{ removed: name }`. Removes the skill from `relay.skills` and uninstalls from all agent paths (deletes the installed file or removes the section for Codex-style agents). Throws `-32603` if name not found.

## New VS Code Commands

### `relay.configureWorkflow`
Manual fallback for teams without a standards doc. Opens a series of VS Code input boxes, one per `WorkflowConfig` field, pre-filled with current values if config exists. Saves on completion. Can also be used to edit any field after initial setup.

Registered in `package.json` as `"Relay: Configure Workflow"`.

### `relay.installWorkflowSkills`
Shows a two-level QuickPick: skills × detected agents. User selects which combinations to install or reinstall. Useful after adding a new agent or after a skill is updated. Calls `SkillRegistry.install(skillName, adapter)` for each selected combination.

Registered in `package.json` as `"Relay: Install Workflow Skills"`.

## AgentAdapter Extension

Two new methods added to the `AgentAdapter` interface:

```typescript
interface AgentAdapter {
  // existing
  readonly id: string;
  readonly label: string;
  isInstalled(): Promise<boolean>;
  isConfigured(port: number): Promise<boolean>;
  configure(port: number): Promise<void>;

  // new
  readonly skillInstallPath: string | null;  // null if agent does not support file-based skills
  isSkillInstalled(skillName: string): Promise<boolean>;
  installSkill(skillName: string, content: string): Promise<void>;
  uninstallSkill(skillName: string): Promise<void>;
}
```

`content` passed to `installSkill` is the skill body with frontmatter stripped — just the markdown the agent should load.

**Install paths and formats per agent:**

| Agent | `skillInstallPath` | Format |
|---|---|---|
| Claude Code | `~/.claude/skills/` | `<name>.md` written as-is |
| Cursor | `.cursor/rules/` (workspace root) | `<name>.mdc` wrapped in Cursor rule format |
| Gemini | `~/.gemini/skills/` | `<name>.md` written as-is |
| Codex | workspace root | Named section appended to `AGENTS.md` |

`JsonFileAdapter` provides a default `skillInstallPath: null` and no-op implementations of the skill methods. Each concrete adapter overrides with its actual path and format.

## AgentRegistry Extension

New method `showSkillInstallPrompt(skills: Skill[])`:

- Called on activation after `showSetupPrompt()` resolves
- Also called by `SkillRegistry` when a new skill is confirmed via `add_skill`
- Checks `relay.workflowSkillDismissed` flag — if set, does nothing (unless called from `relay.installWorkflowSkills` command, which always shows the picker)
- Shows VS Code information message: `"Relay has N workflow skill(s) ready. Install on detected agents?"`  with `"Install"`, `"Not now"`, `"Don't ask again"` options
- On `"Install"`: calls `SkillRegistry.installAll(configuredAdapters)`
- On `"Don't ask again"`: sets `relay.workflowSkillDismissed`

## Confirmation UX

### Config review (`submit_workflow_config`)

1. VS Code information message: `"An agent submitted workflow config. Review before saving?"` with `"Save"` and `"Review first"` buttons.
2. `"Review first"` calls `create_page` internally to render the proposed config as a readable `.relay` page and opens it in the page viewer. A follow-up notification shows `"Save workflow config?"` with a `"Save"` button.
3. `"Save"` (from either path) writes the merged config to `relay.workflowConfig`.
4. Dismissing without saving clears the pending state without affecting the existing config.

### Skill review (`add_skill`)

1. VS Code information message: `"Agent submitted skill '<name>': <description>. Install on all agents?"` with `"Install"`, `"Review first"`, and `"Dismiss"` buttons.
2. `"Review first"` opens the skill markdown in a VS Code text editor tab (untitled, read-only). A follow-up notification shows `"Install '<name>' skill?"` with an `"Install"` button.
3. `"Install"` (from either path) stores the skill and runs installation on all configured agents.
4. `"Dismiss"` clears the pending state.

## Skill Format

Skills submitted via `add_skill` must follow this format:

```markdown
---
name: dev-flow
description: >-
  One-line description — used by agents to decide when to invoke.
  Keep it specific enough to avoid ambiguity with other skills.
triggers:
  - starting a new task
  - reviewing a PR
  - picking up an issue
agents: all
version: 1
---

# Skill body — plain markdown, works across all agents.

At startup, call `get_workflow_config` to read team-specific values
(Slack channels, GitHub org, language, PR account, GitHub username).
```

**Required frontmatter fields:** `name` (kebab-case), `description`
**Optional fields:** `triggers` (list of invocation phrases), `agents` (`all` or a list of agent IDs: `claude-code`, `cursor`, `gemini`, `codex`), `version` (integer, defaults to 1, auto-incremented on resubmit)

**Content rules:**
- No hardcoded values — channel names, usernames, org names, and URLs always read from `get_workflow_config` at runtime
- Reference MCP tools by name only (`get_workflow_config`, `create_page`) — each agent knows how to call them via its own MCP client
- Describe behavior, not implementation details: write "fetch the linked issue" not "run `gh issue view #N`"
- No agent-specific syntax in the shared body — if a section only applies to one agent, put it in a separate skill with `agents: [that-agent-id]`
- `version` increments automatically when a skill with the same `name` is resubmitted

The extension exposes the skill format as an MCP resource at `relay://guide/skill-format` so agents can read it before submitting.

## Workflow Paths (skill content, not extension code)

The generalized dev-flow skill covers seven paths. These live in the submitted skill markdown — the extension does not implement workflow logic.

| # | Path | Entry condition |
|---|---|---|
| 1 | Pick from To Do | Self-assigning from the top 3 unassigned issues |
| 2 | Directly assigned | Issue already has your username as assignee |
| 3 | Own issue pickup | You created the issue and are now picking it up |
| 4 | Continue in-progress | Already on a branch, returning to existing work |
| 5 | Pre-PR self-review | Work done locally, before `commit-push-pr` |
| 6 | AI review loop | Draft PR open, CodeRabbit/AI comments in |
| 7 | Reviewer path | Assigned as reviewer on someone else's PR |

For paths 1–3: detect ≤8h vs >8h estimate, route to direct start or alignment flow.
For path 7: fetch issue + PR via `gh` CLI, call `create_page` with a review brief (requirements, author, approach, how-to-test, files changed). Plain `.relay` page, no special tool.

Paths 5 and 7 call `create_page` for explanations, checklists, and summaries whenever the user asks. Plain `.relay` pages throughout — no special review page type.

`commit-commands:commit-push-pr` is invoked from path 5 to commit, push, and open the draft PR. `co-ops:issue-alignment`, `co-ops:issue-alignment-check`, and `co-ops:pr-merge-check` are invoked from the relevant paths. The skill reads which Slack channels to post to from `get_workflow_config` at runtime.
