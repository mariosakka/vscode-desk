Architecture

UI layer
Webview

Vanilla HTML/CSS/JS running in VS Code's sandboxed webview. Communicates with the extension host via postMessage.

Host layer
PortalViewProvider

WebviewViewProvider that owns the sidebar lifecycle. Bridges webview messages to DataService and pushes updates back.

Data layer
DataService

All reads and writes to VS Code globalState. Exposes typed CRUD methods. No side effects.

Network layer
McpServer

Lightweight http.createServer on 127.0.0.1. Parses JSON-RPC, dispatches to DataService, then calls provider.refresh().

Utility
FaviconService

Fetches favicons over HTTPS with redirect following (max 2 hops). Caches base64 data URLs in globalState.

Data flow — user adds a bookmark via sidebar click

// webview → extension host
webview.postMessage({ type: 'addBookmark', tabId, title, url })

// PortalViewProvider receives it
this._dataService.addBookmark(tabId, { title, url, icon, description })

// DataService writes to globalState, then provider refreshes webview
this._view.webview.postMessage({ type: 'update', data: this._dataService.get() })

Data flow — AI agent adds a bookmark via MCP

// McpServer receives tools/call { name: 'add_bookmark', arguments: { tab_id, title, url } }
const icon = args.icon ?? await this.faviconService.getIcon(args.url)
this.dataService.addBookmark(args.tab_id, { title, url, icon, description })
this.provider.refresh()  // pushes update to open webview

Persistence

All data lives in VS Code's ExtensionContext.globalState under two keys:

KeyTypeContents
relay.dataPortalDataAll tabs and bookmarks. Survives VS Code restarts.
relay.favicon-cacheRecord<hostname, CacheEntry>Base64 favicon data URLs, each with a fetchedAt timestamp. Entries older than 30 days are re-fetched on next access.
globalState is scoped to the extension — it won't interfere with other extensions and is automatically cleared when Relay is uninstalled.

Press F5 in VS Code to launch the Extension Development Host. It automatically runs the compile step via the .vscode/launch.json preLaunchTask.

Testing

Tests run in Node via Jest + ts-jest. The vscode module is mocked at src/__mocks__/vscode.ts, so tests never need a real VS Code instance.

Writing a new test

Each test file sets up its own mockState object and constructs a mockContext with globalState.get/update backed by it. No global setup needed.

const mockState: Record<string, any> = {}
const mockContext = {
  globalState: {
    get: (key: string) => mockState[key],
    update: (key: string, val: any) => { mockState[key] = val },
  },
} as any

Webview Details

Content Security Policy

The webview CSP allows scripts and styles only from VS Code's trusted ${cspSource}, plus img-src data: so base64 favicon data URLs render without external network requests. No http: or https: image sources are permitted.

Theming

Both CSS files map Relay's internal variables (--bg, --text, --accent, etc.) directly onto VS Code's injected theme tokens (--vscode-editor-background, --vscode-textLink-foreground, etc.) with hardcoded fallbacks. No JavaScript theme logic is needed — VS Code re-injects its tokens whenever the user changes theme, and the CSS cascade handles the rest.

Extending Relay

Adding a new MCP tool

  1. Add the tool's JSON schema to the TOOLS array in src/mcp/toolSchemas.ts.
  2. Add a case for it in McpServer.callTool() in src/mcp/server.ts.
  3. Add a test in src/mcp/server.test.ts.

Adding a new VS Code command

  1. Register the command ID in package.json under contributes.commands.
  2. Add the handler function in src/extension.ts.
  3. Register it via vscode.commands.registerCommand() in activate().

Changing the UI

Edit files in src/webview/. Run npm run watch and reload the Extension Development Host window (Ctrl+R inside the host) to see changes instantly.
